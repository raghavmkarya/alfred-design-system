How connections work

Connect in a week, with read-only access

Read-only by default. Every integration is a secure, read-only API token. Alfred reads your data — it never needs write access to start.

OAuth, not credentials. You authorise through each platform's own OAuth flow and revoke any source in one click. No passwords stored.

History comes with it. Up to a year of history syncs on connect, so the memory starts deep instead of empty.

Zero engineering. No pipelines, no warehouse, no data team. Connections go live in Week 1.

06
Governance & write-back

From read to logged: how write-back works

1
Read
Connections are read-only. Alfred ingests and reconciles — it changes nothing.
2
Recommend
It surfaces a decision-ready action with root cause and projected impact.
3
Authorise
A human approves. Encrypted write-back fires only on explicit authorisation.
4
Act
Alfred executes inside your tools — a budget shift, a pause — exactly as approved.
5
Log
Every change is written to a full audit trail: who, what, when and why.

Nothing Alfred does is silent or irreversible. The default state is read-only; write access is granted per decision, never blanket.

08
Data freshness & reliability

If the data goes stale, you'll know

No silent staleness. If a feed lags or breaks, Alfred flags the affected KPI as stale rather than quietly serving an old number.

Honest degradation. When a source is down, Alfred narrows its confidence and says so — it never presents a partial picture as complete.

Continuous reconciliation. Sources re-sync on a schedule, so the Unified KPI Cockpit shows one reconciled state — not ten disagreeing tabs.

How Alfred surfaces data health
Google Ads · synced 4m agoFresh
HubSpot · syncing nowDelayed
Amazon Ads · token expiredExcluded

Stale sources are excluded from recommendations until they recover — never silently averaged in. Silent stale data is a trust failure, so Alfred refuses to hide it.

Your data is isolated, with a private pattern library built for your organisation alone.
